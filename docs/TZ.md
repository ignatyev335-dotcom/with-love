# With Love

> **RSVP-приглашения на свадьбу нового поколения**  
> Полноценная SaaS-платформа: красивые шаблоны → конструктор → публикация → управление гостями в реальном времени.

---

## 🎯 О проекте

**With Love** — это сервис, где организаторы свадеб создают красивые, современные и функциональные сайты-приглашения с RSVP.

Гости получают элегантную страницу с таймером, историей, программой дня, локацией, формой подтверждения присутствия, пожеланиями, музыкой и всей необходимой информацией.

### Модель продукта
- **Freemium** + разовые платежи за премиум-шаблоны, дополнительные функции и снятие ограничений
- **1 организатор = 1 активная свадьба** (упрощает логику и данные)
- Публичные ссылки вида `withlove.app/invite/unique-slug` (для MVP достаточно)

Платформа включает:
- Мощный конструктор приглашений
- Галерею из 40+ премиальных шаблонов в разных стилях
- Личный кабинет организатора
- Полноценную админ-панель платформы
- Фоновая музыка на опубликованных приглашениях
- Полноценную поддержку i18n (русский + английский)

**Ключевой принцип**: Один сервис — безупречный результат. Готовый сайт за 10–15 минут.

### Почему именно этот стек?
- Максимально дружелюбен к **vibe coding** / Cursor / AI-ассистентам
- Vercel + Supabase = деплой за 30 секунд
- Realtime из коробки (идеально для живых RSVP и счётчиков гостей)
- Полноценный TypeScript с отличным автодополнением
- Легко масштабировать в будущем

**Альтернатива** (если не хочешь Supabase): Prisma + PostgreSQL (Neon) + Clerk (auth). Но Supabase выигрывает по скорости разработки.

## 🔄 Основные пользовательские потоки

### 1. Организатор (главный пользователь)
1. Лендинг → «Создать приглашение»
2. Выбор стиля / шаблона (или начать с нуля)
3. Регистрация / вход
4. **Конструктор**:
   - Включает/выключает блоки
   - Редактирует текст, фото, цвета (в рамках стиля)
   - Меняет порядок блоков
   - Предпросмотр в реальном времени (desktop + mobile)
5. Публикация → получает уникальную ссылку + QR-код
6. Делится ссылкой с гостями
7. В личном кабинете:
   - Видит все ответы в реальном времени
   - Управляет гостями (статусы, +1, рассадка)
   - Отправляет напоминания
   - Смотрит статистику

### 2. Гость
1. Открывает красивую ссылку
2. Смотрит всю информацию (таймер, программа, локация, дресс-код и т.д.)
3. Заполняет простую форму RSVP (имя, присутствие, количество гостей, пожелания, диетические ограничения)
4. Видит подтверждение + опционально — список подтвердивших гостей

### 3. Администратор платформы
- Управление пользователями и подписками
- Модерация шаблонов и контента
- Финансы и выплаты
- Поддержка клиентов
- Аналитика платформы

## 🗄 Предлагаемая схема базы данных (Supabase / Drizzle)

**Ключевые таблицы** (упрощено из-за правила «1 организатор = 1 свадьба»):

- `profiles` — профиль организатора (связан 1:1 с `weddings`)
- `weddings` — событие (свадьба) — у каждого пользователя максимум одна активная
- `invitations` — опубликованное приглашение (JSON-конфиг блоков + метаданные, slug, published status)
- `templates` — готовые шаблоны
- `guests` — список гостей конкретной свадьбы
- `rsvps` — ответы гостей
- `payments` — разовые платежи
- `wishes` — пожелания гостей
- `music_tracks` — загруженная/выбранная музыка для приглашения

**Гибкость конструктора**:
Храни конфигурацию приглашения в поле `config` типа `jsonb`. Это позволяет легко добавлять новые блоки без миграций.

Поскольку у пользователя только одна свадьба — логика сильно упрощается (меньше joins, проще права доступа).

Пример структуры `invitation.config`:
```json
{
  "blocks": [
    { "type": "hero", "data": { ... } },
    { "type": "story", "data": { ... } },
    { "type": "schedule", "data": { ... } },
    ...
  ],
  "theme": "classic" | "minimal" | "luxury" | "travel",
  "colors": { ... },
  "fonts": { ... }
}
```

## 📋 Инструкции для ИИ-кодера (ОБЯЗАТЕЛЬНО ЧИТАТЬ)

**Ты — senior frontend + fullstack разработчик, специализирующийся на красивых SaaS-продуктах.**

### Главные правила:

1. **Визуальное соответствие референсам — №1 приоритет**
   - Все 14 предоставленных скриншотов — это истина.
   - Каждый экран должен выглядеть **максимально близко** к референсу (отступы, цвета, типографика, расположение элементов, стиль кнопок, карточек, иконок).
   - Если сомневаешься — делай **точно как на референсе**.

2. **Работай маленькими, атомарными шагами**
   - Сделал лендинг → `git commit -m "feat: реализовал главный лендинг по референсу 01"` → `git push`
   - Сделал авторизацию → commit + push
   - Каждый значимый экран / фича = отдельный коммит + push

3. **GitHub + Vercel**
   - Репозиторий подключён к Vercel.
   - **Каждый push в main** автоматически деплоится.
   - После каждого пуша проверяй, что деплой прошёл успешно.

4. **Качество кода**
   - TypeScript везде (strict mode)
   - Компоненты переиспользуемые и хорошо типизированы
   - Server Actions для мутаций (где возможно)
   - Красивые, семантические имена переменных и функций

5. **i18n (интернационализация)**
   - Используй `next-intl`
   - Поддержка минимум двух языков: **Русский** (по умолчанию) + **English**
   - Все тексты в интерфейсе, на лендинге, в дашборде и на публичных приглашениях должны быть переведены
   - Структура: `messages/ru.json` и `messages/en.json`

6. **Адаптивность**
   - Все экраны должны идеально выглядеть на мобильных, планшетах и десктопе (см. референс 03).

6. **Realtime**
   - Используй Supabase Realtime для:
     - Обновления счётчиков RSVP
     - Live-обновления списка гостей в кабинете организатора

7. **Музыка на приглашении**
   - На опубликованной странице приглашения должна играть фоновая музыка (по выбору организатора)
   - Используй нативный `<audio>` элемент + хороший UX (кнопка Play/Pause + опциональное автозапуска после первого взаимодействия пользователя)
   - Организатор должен иметь возможность загрузить свой трек или выбрать из небольшой библиотеки
   - Автозапуск музыки должен быть мягким и не агрессивным (браузеры блокируют агрессивный autoplay)
